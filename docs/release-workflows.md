# Release Workflows

This repository now uses a manual, auditable release process driven by `.github/workflows/release-orchestration.yml`. npm publishing for the generic workspace packages, `typst.node`, and projects runs inside that workflow with npm trusted publishing, while Docker image and Rust crate publishing remain separate reusable workflows.

## Protected npm environment

- GitHub Environment: `npm-publish`
- Trusted publishing runtime: Node `22.14.0` or newer with npm `11.5.1` or newer
- npm credentials: GitHub OIDC with `id-token: write`
- npm registry: release workflows set npm and Yarn registry environment to `https://registry.npmjs.org/`
- npm token handling: the workspace package, `typst.node`, and project npm publish jobs do not use `NPM_TOKEN` or write a registry token into `.npmrc`
- npm provenance metadata: published package manifests set `repository.url` to `https://github.com/Myriad-Dreamin/typst.ts`
- npm dry-run reruns: verification steps set `NPM_CONFIG_FORCE=true` only for npm dry-run publish commands so reruns can pass when the version already exists on npm
- npm live publish reruns: publish jobs skip package versions that already exist on npm, while other npm publish failures still fail the job

## Workflow coverage

### `.github/workflows/release-orchestration.yml`

Manual workflow for one full release run per version. This is the top-level trusted publishing workflow filename for the covered npm packages and projects. It does not expose partial package or project selection.

The fixed release order is:

1. verify the generic workspace npm packages
2. verify the publishable project batch
3. publish the generic workspace npm packages
4. call `.github/workflows/release-node.yaml`
5. publish the npm and Cargo project batch
6. call `.github/workflows/docker.yaml`
7. call `.github/workflows/reelase-crates.yaml`

The npm package scope is intentionally not duplicated in this document. The
authoritative release scope is encoded by the filters, publish commands, and
package manifests used by the workflow jobs. The workflow summaries report which
entries were included or skipped for a concrete run.

The generic workspace and npm project publish jobs use
`scripts/npm-publish-if-needed.mjs` so reruns skip package versions that are
already present on npm without masking unrelated publish failures.

### `.github/workflows/release-node.yaml`

Dedicated `typst.node` release workflow. It is only callable from `release-orchestration.yml` through `workflow_call` and preserves the existing napi-rs build, test, artifact preparation, npm publish, and GitHub Release upload behavior for the packages generated by that release path.

This workflow also uploads the per-platform `.node` assets to the GitHub Release tag supplied to the workflow. Its npm publish job uses the protected `npm-publish` environment and GitHub OIDC trusted publishing. If an npm package version was already published, the publish wrappers treat that package as complete and continue so the remaining release jobs are not blocked by a rerun; other npm publish failures still fail the job.

The npm trusted publisher registration for these packages uses `release-orchestration.yml`, because npm validates the caller workflow when a reusable workflow performs the publish. Treat `.github/workflows/release-orchestration.yml` as the live release entry point for `typst.node`.

### `.github/workflows/docker.yaml`

Manual and reusable Docker publish workflow for the repository image on GHCR. It publishes:

- the repository image on the configured platforms

Authentication uses `GITHUB_TOKEN` with `packages: write`.

### `.github/workflows/reelase-crates.yaml`

Manual and reusable Rust workspace crate workflow. It verifies and publishes the
crate set encoded by the workflow.

Authentication uses `CRATES_IO_TOKEN`.

## npm trusted publisher checklist

Detailed setup steps are documented in
[`npm-trusted-publishing-setup.md`](./npm-trusted-publishing-setup.md).

Before the first live npm publish for a package:

1. Create or update the protected GitHub Environment named `npm-publish` with the required reviewers and any branch or tag restrictions you want for live publish approvals.
2. In npm package settings, add a trusted publisher entry for the exact workflow file that publishes that package.
3. Confirm that the trusted publisher entry points at this repository and the `npm-publish` environment expectations match the workflow.
4. Keep the workflow filename stable after registration, or update the npm trusted publisher entry before the next release.

Register this workflow file for every covered trusted-publishing package:

- GitHub workflow file: `.github/workflows/release-orchestration.yml`
- npm trusted publisher `Workflow filename` value: `release-orchestration.yml`
- Covered packages: derive from the current release workflow and package manifests instead of copying a package list into this document.

## Credential and approval boundaries

- generic workspace npm package publishing, `typst.node` npm publishing, and npm project publishing: protected `npm-publish` environment plus GitHub OIDC with `id-token: write`
- `typst.node` GitHub Release asset uploads: `GITHUB_TOKEN` with `contents: write`
- GHCR Docker publishing: `GITHUB_TOKEN` with `packages: write`
- Rust workspace crates and Cargo project publishing: `CRATES_IO_TOKEN`
